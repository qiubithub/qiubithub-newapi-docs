# 渠道管理

## 为什么会有"渠道管理"这个设计？

### 关于稳定性的真相

**底层渠道资源天然是不稳定的。**

为了帮大家把价格打下来，我们通过技术手段把官方的"大包"拆成了"小包"分给大家。这相当于实现了 **100% 的套餐利用率**。

但这对官方来说是亏本生意，所以一定会触发"风控"。

**风控 = 不稳定**

这是为了极致性价比必须付出的代价，而不是我们故意把便宜的渠道做烂。

为了解决这个问题，我们不能把鸡蛋放在一个篮子里，所以设计了 **"多渠道混合"** 机制。

---

## 我们是怎么帮您省钱的？

为了不让大家用着用着突然断连，我们接入了多种底层资源：
- 官方订阅渠道
- AWS 渠道
- 特价渠道
- 备用渠道

### 系统的自动执行逻辑

我们将策略简化为八个字：**"价格优先，自动兜底"**。

**💰 优先省钱**
只要您开着特价渠道，系统永远优先调用最便宜的那个。

**🛡️ 自动容灾**
如果便宜的渠道因为风控"挂了"，系统会**毫秒级**自动切换到稍贵的备用渠道，保证你的代码生成任务不报错。

**🔄 自动回切**
一旦检测到便宜渠道恢复（风控解除），系统会自动切回去，最大力度帮您省钱。

> **总结就是：** 你只管开着，系统会想办法让你用最便宜的；便宜的挂了，系统会自动调个贵的顶上，等便宜的好了再切回来。

---

## 既然全自动，为什么还要允许我"手动关闭"？

虽然便宜的渠道性价比高，但有时候您可能会感觉"降智"：
- 感觉没有官方渠道那么聪明
- 在特定复杂任务上表现不如意
- 对响应质量有极高要求

这时候，选择权在您手里：

### 应该怎么选？

| 场景 | 推荐配置 | 说明 |
|------|---------|------|
| **追求极致性价比**（默认推荐） | 把"特价渠道"和其他渠道**都开着** | 让系统自己去调度，能省则省 |
| **对质量要求极高**（不差钱） | 关闭"特价渠道"，只保留官方或 AWS 渠道 | 哪怕多花点钱，也不想体验因为风控带来的波动 |

## 如何操作

操作非常简单，仅需三步：

### 1. 选择模型商

在左侧导航栏选择你要配置的模型商（如 `Anthropic`、`OpenAI`、`Google`）

### 2. 启用或禁用渠道

在右侧列表中，点击开关启用或禁用特定渠道。

### 3. 保存配置

:::danger 别忘了点保存！
修改完开关配置后，**必须点击右上角的"保存"按钮**。如果不点保存，所有配置都不会生效！
:::

## 渠道类型说明

| 渠道类型 | 特点 | 价格 | 稳定性 | 推荐场景 |
|---------|------|------|--------|---------|
| **官方订阅** | 官方直接提供 | 最高 | 最高 | 对质量要求极高的生产环境 |
| **AWS 渠道** | 通过 AWS Bedrock | 较高 | 高 | 需要稳定性的企业用户 |
| **特价渠道** | 特价资源 | 最低 | 中等 | 追求性价比的开发测试 |
| **备用渠道** | 应急使用 | 中 | 高 | 自动兜底 |

## 最佳实践

### 日常使用
- ✅ **保持所有渠道开启**：让系统自动选择最优渠道
- ✅ **关注渠道状态**：定期查看渠道健康度
- ✅ **设置备用渠道**：确保至少有一个高稳定性渠道处于开启状态

### 问题排查
- 🔍 **感觉"降智"**：尝试关闭特价渠道，使用官方渠道
- 🔍 **频繁报错**：检查是否有渠道被风控，暂时关闭问题渠道
- 🔍 **成本过高**：开启更多特价渠道，让系统自动选择

### 企业用户建议
- 生产环境：主要使用官方或 AWS 渠道
- 开发测试：可以使用特价渠道
- 设置监控：关注各渠道的可用性和响应质量

## 常见问题

### Q：为什么有时候特价渠道会自动关闭？

A：当特价渠道触发风控或不可用时，系统会自动标记为不可用。风控解除后会自动恢复。

### Q：渠道切换会影响正在进行的对话吗？

A：不会的。渠道切换只影响新的 API 请求，已建立的连接不受影响。

### Q：如何知道当前使用的是哪个渠道？

A：可以在 API 响应的 Header 中查看 `X-Channel-Used` 字段，或在控制台查看调用日志。

### Q：可以同时开启多个渠道吗？

A：可以。推荐同时开启多个渠道，让系统自动选择最优渠道。
